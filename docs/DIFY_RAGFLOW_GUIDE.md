# Dify 与 RAGFlow 配合开发指南

## 1. 职责划分

- RAGFlow：用于资料清洗、政策知识卡整理、文档检索与人工校验。
- Dify：用于编排问答工作流，把用户问题、租户信息、检索结果和风险提示组织成易懂回答。
- 本项目后端：负责认证、租户隔离、日志、反馈、安全脱敏、FAQ 兜底和统一 API。

## 2. 选型优势

- Dify 的优势在于工作流编排和 Prompt 治理，适合把模型选择、检索策略、回答格式和风险提示从业务代码中抽离出来。
- RAGFlow 的优势在于资料预处理和知识治理，适合对政策原文、网页资料和企业制度做清洗、切分、复核和检索验证。
- 两者配合后，业务后端只负责稳定 API、权限、日志和兜底回答；AI 能力可以独立调参、替换模型或调整知识库，不影响核心业务表。
- 对合规问答场景而言，这种分工能保留来源追溯和人工复核流程，降低直接把原始资料交给大模型造成的误答风险。

## 3. Dify 工作流输入

后端调用 Dify 时会传入：

| 字段 | 说明 |
| --- | --- |
| query | 用户问题，已做基础脱敏 |
| user | 用户标识 |
| tenant_code | 租户编码 |
| tenant_name | 租户名称 |
| region | 租户地区 |
| answer_style | 回答风格要求 |

建议工作流包含：

1. 问题分类：劳动合同、工资、社保、医保、假期、仲裁、数据安全。
2. 风险识别：低、中、高。
3. 知识检索：优先检索租户专属资料，再检索陕西公共知识包。
4. 回答生成：结论、适用条件、来源依据、办理路径、风险提示。
5. 结构化输出：方便后端和前端展示来源、行动项和风险等级。

## 4. 回答质量要求

- 先给结论，再说明适用条件。
- 涉及金额、日期、缴费基数等易变信息必须提示“以官方最新口径为准”。
- 不要在回答中暴露身份证号、手机号、银行卡号、客户名单或合同价格。
- 对政策不确定项明确标注“待复核”，不要编造具体数值。
- 对高风险问题建议 HR/法务复核。

## 5. RAGFlow 资料整理建议

资料导入前先整理为知识卡：

```text
标题：
分类：
适用地区：
适用对象：
政策结论：
适用条件：
办理路径：
所需材料：
风险提示：
官方来源：
发布日期：
执行日期：
更新时间：
复核状态：
```

不建议直接把原始网页、截图和含个人信息的企业材料导入知识库。

## 6. 本地服务状态

当前本机服务探测结果在后台“平台概览”展示：

- Dify Web：`http://127.0.0.1/`
- Dify API：`http://127.0.0.1/v1`
- RAGFlow Web：`http://127.0.0.1:8880/`
- RAGFlow API：`http://127.0.0.1:9380`
- Dify 容器内访问 RAGFlow 外部知识库 API：`http://host.docker.internal:8880/api/v1/dify`

如果 Dify API Key 为空，后端会自动走本地 FAQ 兜底。
