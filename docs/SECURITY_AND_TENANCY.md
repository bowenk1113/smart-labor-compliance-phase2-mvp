# 安全与多租户设计说明

## 1. 租户隔离模型

平台采用共享数据库、共享表、行级租户隔离模式：

- `slc_tenants` 保存租户元数据
- 业务表通过 `tenant_id` 关联租户
- 后台依赖 JWT 中的管理员角色和租户绑定做强制过滤
- 用户端依赖 `X-Tenant-Code` 解析租户，不允许跨租户读取历史记录

这种方式适合 MVP 和中小规模 SaaS。若进入强监管或大型私有化场景，可升级为单租户独立数据库。

## 2. 安全技术选型优势

- `JWT` 适合前后端分离应用，后端无需保存会话状态，便于水平扩展和多端调用。
- `bcrypt` 具备密码哈希加盐和计算成本控制能力，比明文或普通摘要更适合保存后台账号密码。
- `tenant_id` 行级隔离能在共享数据库中控制数据范围，成本低、迁移快，适合 MVP 和中小规模 SaaS。
- `X-Tenant-Code` 让用户端请求可以明确归属租户；后台则依赖 JWT 中的角色和租户绑定，避免仅信任客户端传参。
- 敏感信息脱敏、请求体大小限制、限流和安全响应头集中放在后端，有利于所有前端页面复用同一套安全边界。

## 3. 角色权限

| 角色 | 权限 |
| --- | --- |
| super_admin | 全平台租户、账号、数据、知识包、日志 |
| tenant_admin | 本租户账号、FAQ、来源、日志、反馈、知识包 |
| operator | 本租户 FAQ、来源、日志、反馈、知识包 |
| viewer | 本租户概览与日志只读 |

## 4. 数据防泄露措施

- 敏感字段写入日志前脱敏：身份证号、手机号、银行卡号、邮箱
- 密码只保存 bcrypt 哈希
- JWT 使用 `.env` 中的 `JWT_SECRET_KEY` 签名
- API 响应不返回密码哈希、Dify API Key 等敏感配置
- CORS 限制为本机前端地址
- 基础请求频控防止接口被刷
- 请求体大小限制降低异常大包攻击风险

## 5. 并发与响应能力

后端使用 FastAPI + Uvicorn，数据库连接池配置：

- `pool_size=10`
- `max_overflow=20`
- `pool_pre_ping=True`
- `pool_recycle=1800`

部署时可使用多 worker：

```bash
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 2
```

在 8GB 内存设备上不建议开太多 worker，优先保持 Dify/RAGFlow/MySQL 的内存余量。

## 6. 生产上线检查

- 修改 `JWT_SECRET_KEY`
- 修改初始管理员密码
- 限制 MySQL 对外暴露范围
- Dify/RAGFlow API Key 使用环境变量，不写入代码库
- 官方政策来源做二次复核，移除“演示口径”
- 配置 HTTPS 和反向代理安全头
- 对日志保留周期和导出权限做制度约束
